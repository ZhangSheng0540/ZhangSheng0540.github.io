mysql相关

B+树和B树的区别
  • B树的每个节点存储了key和data，而B+树只在叶子节点存储data数据
  • B+树的每个叶子节点的指针指向相邻的叶子节点，构成一个有序链表，而B树叶子节点指针为null

多线程

线程安全的几个特性
  • 原子性，正在执行的操作不对被其他线程干扰，一般使用cas操作和unsafe 关键字
  • 可见性，在一个线程中定义一个变量，他的状态会立马被其他线程知晓，一般使用volatitle关键字
  • 有序行，保证线程内的串行语句，避免指令重排，一般使用Synchronized，ReentrantLock，volatitle关键字
线程的几种状态
  • 初始状态，刚创建好线程时的状态
  • 运行状态，线程调用了start()方法，表示该线程处于就绪状态，线程获取到了cpu时间片，由可运行状态到了运行状态
  • 阻塞状态，等待获取锁让线程进入阻塞状态。
  • 等待转态，进入该状态的线程需要等待其他线程做出一些特定动作（通知或中断）
  • 超时等待，该状态不同于WAITING，它可以在指定的时间后自行返回
  • 死亡状态，run()方法的正常退出就让线程进入到死亡状态，还有当一个异常未被捕获而终止了run()方法的执行也将进入到死亡状态
可重入锁（递归锁）

是指在同一个线程在外层方法获取锁的时候，再进入该线程的的内层方法会自动获取锁（前提，锁对象是同一个对象），不会应为之前获取过锁还没释放而等待
种类：

  • 隐式锁 （Synchronized关键字使用的锁）
  • 显示锁（Lock,即ReentrantLock这样的可重入锁）

ReentrantLock和Synchronized区别

使用方式

  • Synchronized可以修饰类，方法，代码块，自动释放锁
  • ReentrantLock一般在try——catch——finally语句中，在try语句中获取锁，在finally语句中释放做，手动释放锁。

可中断的

  • Synchronized不可中断
  • ReentrantLock提供不可中断和可中断两种方式，其中lockInterruptibly()方法表示可中断，lock方法表示不可中断

公平和非公平

  • Synchronized是非公平锁
  • ReentrantLock提供公平和非公平两种方式，默认是非公平的，在构造方法中添加true属性可以构造为公平锁。

实现机制

  • Synchronized是通过monitorrenter和monitorexit这两个字节码制定操作的。
  • ReentrantLock

锁定多个条件condition

  • Synchronized没有
  • ReentrantLock可以精确唤醒
  • Synchronized JVM层面 关键字，不需要用户手动释放。
  • ReentrantLock 具体类 api层面，手动释放。
线程池的好处

线程池的主要作用是控制运行的线程的数量，处理过程中将任务放到对列，然后在线程创建后启动这些任务，如果线程数量超过了最大数量，超出数量的线程排队等候，等其他线程执行完毕，再从对列中取出任务执行。

线程池的主要特点：线程复用，控制最大并发数，管理线程

优势：

  • 降低资源消耗。通过重复利用已经创建的线程降低线程创建和销毁造成的消耗
  • 提高响应速度。但任务到达时，任务可以不需要等待线程创建就能执行
  • 提高线程的可管理性。线程是稀缺资源，如果无限制的创建，不仅会消耗线程资源，还会降低系统的稳定性，使用线程池可以进行统一的分配，调优和监控。

ThreadPoolExector中的7个参数：

  • corePoolSize: 今日当值线程，线程池中常驻核心线程数
  • maximumPoolSize: 线程池能容纳的同时执行的最大线程数
  • keepAliveTime: 多余空闲线程的存活时间
  • until：keepAliveTme的单位
  • workQueue: 阻塞对列
  • threadFactory: 线程 工厂
  • handle：拒绝策略

线程池执行过程：

  • 1、在创建了线程池后，等待提交过来的任务请求。
  • 2、当调用execute()方法添加一个请求任务时，线程池会做一下判断：
    1. 如果正在运行的线程数量小于corePoolSize，那么马上创建线程运行这个任务；
    1. 如果正在运行的线程数量大于或等于corePoolSize,那么将这个任务放入对列；
    1. 如果这个时候对列满了且正在运行的线程数量小于maximumPoolSize,那么创建非核心线程立刻运行这个任务；
    1. 如果对列满了且正在运行的线程数量大于或等于maximumPoolSize，那么线程池会启动饱和拒绝策略来执行。
  • 3、当一个线程完成任务时，他从对列中取下一个任务来执行；
  • 4、当一个线程无事可做一段时间（keepAliveTime）时，线程会判断：
    1. 如果当前运行的线程数量大于corePoolSize,这个线程会别停掉；
    1. 线程池的所有任务完成后他最终会收缩到corePoolSize大小；

4种拒绝策略：

  • AbortPolicy(默认)：直接抛出RejectExecutionExceptio异常，阻止系统正常运行
  • CallerRunPolicy: 调用者运行，该策略不会抛弃任务，也不会抛出异常，而是将某些任务回退给调用者，从而降低新任务的流量
  • DiscardOldPolicy: 抛弃对列中任务等待最久的任务，然后把当前任务加入到对列中尝试再次提交当前任务
  • DiscardPolicy： 直接丢弃任务，不予处理也不抛出异常，如果允许任务丢失，这是最好的一种方案。

如何配置合理的线程数：

  • CUP密集型，cup密集型会进行大量的计算，所以线程数maximumPoolSize大小一般设置为cup核心数+1
  • IO密集型,io密集型会进行大量的io操作，会产生阻塞，所以线程数maximumPoolSize大小一般设置为cup/1-阻塞系数（0.8–0.9之间）

JVM相关

JVM结构
  1. 类加载器（Class Loader）: 就是将Class文件加载到内存，把描述类的数据从Class文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可以被虚拟机直接使用的Java类型
  2. 运行时数据区(Run Data Area) : JVM管理的内存了
  3. 执行引擎（Execution engine）: 执行引擎用于执行指令，不同的java虚拟机内部实现中，执行引擎在执行Java代码的时候可能有解释执行（解释器执行）和编译执行（通过即时编译器产生本地代码执行
  4. Native interface 和 Native libraries : 是其它编程语言交互的接口。当调用native方法的时候，就进入了一个全新的并且不再受虚拟机限制的世界，所以也很容易出现JVM无法控制的native heap OutOfMemory
JVM内存模型
  1. 程序计数器，线程私有，当前线程所执行的字节码的行号指示器，用于记录正在执行的虚拟机字节指令地址
  2. 虚拟机栈，线程私有，每一条线程都有自己私有的 Java 虚拟机栈
  3. 本地方法区 ，线程私有，只不过它服务于Native方法
  4. java堆，线程共享，GC的主要活动区域
  5. 方法区，线程共享，java1.8中的元空间，元空间放置于本地的内存中，包括常量池，字段、方法数据、构造函数和普通方法的字节码内容，还包括一些在类、实例、接口初始化时用到的特殊方法
垃圾回收算法
  • 引用计数器， 判断一个对象有没有被引用，缺点是没办法解决循环引用问题。
  • 复制拷贝算法,将没有被回收的放到一块区域,然后把另一块区域清除，缺点是浪费了一块内存空间。
  • 标记清除算法，将没有被引用的对象标记出来，然后回收这些没有被引用的对象，缺点是产生了内存碎片。
  • 标记整理算法，将存活的对象往一段移动，然后清理以外的区域
  • 分代收集算法，把java堆分为新生代和老年代，新生代用复制算法，老年代用标记清楚或标记整理算法。
垃圾收集器
  • Serial: 串行回收，会产生stw
  • Paraller: 并行回收,会产生stw
  • GMS: 并发回收,会抢占应用的cpu
  • G1: 将堆内存分割成不同的区域region，然后并发的对其进行垃圾回收
  • ZGC:
新生代和老年代和垃圾收集器

新生代:

  1. 串行GC(Serial/Serial Copying)
  2. 并行GC（ParNew）
  3. 并行回收GC(Paraller/ Paraller Scavenge)

老年代:

  1. 串行GC(Serial old/Serial MSC)
  2. 并行GC（ParallerOld/ araller MSC）
  3. 并发标记清除GC(GMS)
可以作为GC Roots的对象
  • 虚拟机栈（栈帧中的局部变量区，也叫做局部变量表）中引用的对象
  • 方法区中的类静态属性引用的对象
  • 方法区中的常量引用的对象
  • 本地方法栈中JNI(Native方法)引用的对象
JVM常用参数
  • -Xms: 初始内存大小，默认物理内存的1/64
  • -Xmx：最大分配内存，默认物理内存的1/4
  • -Xss：单个线程栈的大小，默认512k~1024K
  • -Xmn: 设置年轻代大小
  • -XX:MetaspaceSize: 元空间大小
  • -XX:+PrintGCDetails: GC日志的详细信息
  • -XX:SurvivorRatio: 新生代中eden和s0/s1的比例，默认8:1:1
  • -XX:NewRation: 配置新生代和老年代的比例，默认1:2
  • -XX:MaxTenuringThreshold: 这是垃圾的最大年龄

Spring相关

Spring循环依赖

解决方法：

  1. 换位setter注入方式并且是单例的。
  2. 只有单例的bean会通过三级缓存提前暴露来解决循环依赖的问题，而非单例的bean，每次从容器中获取的是一个新的对象，都会重新创建， 所以非单例的bean是没有缓存的，不会放到三级缓存中。

Spring事务传播机制

支持当前事务

  1. PROPAGATION REQUIRED(默认): 如果当前存在事务，则使用该事务。 如果当前没有事务，则创建一个新的事务。
  2. PROPAGATION_SUPPORTS : 如果当前存在事务，则使用该事务，如果当前不存在事务，则仍以非事务的方式运行。
  3. PROPAGATION_MANDATORY : 如果当前存在事务，则使用该事务，如果当前事务不存在则抛出异常
    不支持当前事务
  4. PROPAGATION_REQUIRES_NEW : 总是创建一个新的事务，如果当前有事务，则将当前事务挂起
  5. PROPAGATION_NOT_SUPPORTED ：非事务的方式运行，如果当前有事务，则将当前事务挂起
  6. PROPAGATION_NEVER : 以事务的方式运行，如果当前有事务，则抛出异常
    嵌套事务
  7. PROPAGATION_NESTED : 当前存在事务，则创建一个嵌套事务来运行，如果当前没有事务，行为等同于PROPAGATION_REQUIRED

Redis内存淘汰策略

reids 过期删除策略

  • 定期删除,对cpu不友好，用处理器性能换取储存空间（时间换空间）
  • 惰性删除，不会立马删除，等下次查询的时候过期了再删除，对内存不友好，用存储空间换取处理器性能，（空间换时间）
  • 定期删除，没隔一段时间执行删除过期键值操作，并通过限制删除操作的执行时长和频率来减少删除操作对cpu的影响（随机抽查，重点抽查）
  • LRU算法：淘汰最近最少使用数据
  • LFU算法：淘汰使用频率最低数据
  • FIFO算法: 淘汰最早数据数据

内存淘汰策略

  • noeviction:不会驱逐任何key
  • allkeys_lru: 对所有的key使用LRU算法进行删除
  • volatile-lru: 对所有设置了过期时间的key使用LRU算法进行删除
  • allkeys_random: 对所有key随机删除
  • volatile_random: 对所有设置了过期时间的key随机删除
  • volatile_ttl: 删除马上要过期的key
  • allkeys_lfu： 对所有的key使用LFU算法进行删除
  • volatile_lfu: 对所有设置了过期时间的key使用LFU算法进行删除
缓存雪崩

原因： 在一个时间段内缓存大面积过期失效，同时有大量请求过来查询，不走缓存，直接查询数据库。

解决方法：

  1. 每个key的失效时间加上随机值
  2. 设置缓存永远不过期
缓存击穿

原因： 对一个 key大量访问，当这个key过期失效后，大量请求打到数据库

解决方法:

  1. 设置缓存永远不过期
  2. 查询数据库的时候加锁
缓存穿透

原因： 用一个在缓存和数据库都不存在的key，不断的去请求，请求会直接打到数据库，有可能击垮数据库

解决方法：

  1. 将数据库中没查到的值放到redis中，设置key的value值为null
  2. 对单个IP值做访问次数限制，
  3. 布隆过滤器
